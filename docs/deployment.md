# 배포 가이드 (ATLAS by MKHZ)

이 문서는 첫 공개 운영 버전을 `atlas.mkhz.kr`에 배포하기 위한 안내다.
**클로드가 직접 할 수 없는 단계(계정·도메인·DNS 작업)는 두목님이 직접 진행해야 한다** — 아래에 정확히 어디부터 어디까지인지 표시했다.

---

## 0. 사전 준비물 (코드 쪽은 완료됨)

이번 작업으로 다음 파일이 준비됐다:

| 파일 | 역할 |
|---|---|
| `index.html` | SEO 메타태그(title/description/canonical/OG/Twitter), GA4 스니펫 추가됨 |
| `robots.txt` | 크롤링 전체 허용 |
| `sitemap.xml` | 현재 단일 페이지 1개 항목 |
| `favicon.svg` | 나침반 심볼 (사진 아님, SVG) |
| `404.html` | "지도를 찾을 수 없습니다" + 역사 탐험 계속하기 버튼 |

**아직 두목님이 채워야 할 것:**
- `og:image` 메타 태그 (현재 주석 처리됨) → 실제 지도 스크린샷을 찍어 `/assets/og-image.png`로 올린 뒤 주석 해제

> GA4 측정 ID(`G-9C05WN48C4`)는 이미 `index.html`(근대·근현대 둘 다)과
> 자료실 정적 페이지 생성기에 적용 완료됐다. 검색어·연도 이동·시대
> 이동·필터·지도 조작·사건 카드 클릭까지 `js/analytics.js`를 통해
> 이벤트로 잡히도록 구현돼 있다 — 더 손댈 것 없음.

## 2. Cloudflare Pages 배포 (두목님이 직접)

이 프로젝트는 번들러·빌드 도구가 없는 **순수 정적 사이트**다 (`npm run build` 같은 빌드 단계 자체가 없다 — HTML/CSS/JS 파일을 그대로 서빙하면 끝). 그래서 Cloudflare Pages의 "빌드 명령어 없음(No build command)" 옵션으로 배포하면 된다.

1. [Cloudflare 대시보드](https://dash.cloudflare.com) 로그인 (mkhz.kr을 이미 관리 중이라면 같은 계정)
2. **Workers & Pages** → **Create application** → **Pages** → **Upload assets** (Git 연동 없이 폴더를 직접 업로드하는 방식 — 가장 단순함) 또는 GitHub 저장소 연동 방식 중 선택
   - **직접 업로드 방식**: 이 프로젝트 폴더(`chronicle/`) 전체를 압축 없이 그대로 업로드. 빌드 명령어는 입력하지 않음, 출력 디렉터리는 루트(`/`)
   - **Git 연동 방식**(권장, 추후 수정이 잦다면): GitHub에 이 폴더를 푸시한 뒤 Cloudflare Pages에서 그 저장소 연결. Build command는 비워둠, Build output directory는 `/`
3. 배포 완료되면 `*.pages.dev` 형태의 임시 주소가 발급됨 — 이 주소로 먼저 **3단계 검수**를 진행

---

## 3. atlas.mkhz.kr 도메인 연결 (두목님이 직접)

mkhz.kr이 이미 Cloudflare에서 관리되고 있다는 전제:

1. Cloudflare Pages 프로젝트 → **Custom domains** → **Set up a custom domain**
2. `atlas.mkhz.kr` 입력
3. Cloudflare가 mkhz.kr의 DNS를 이미 관리하고 있다면, CNAME 레코드를 **자동으로** 추가해준다 (별도로 DNS 화면에 가서 직접 만들 필요 없음 — Pages가 처리)
4. HTTPS는 Cloudflare가 자동 발급·적용한다 (수동 설정 불필요, 다만 전파에 몇 분~몇 시간 걸릴 수 있음)
5. DNS 전파 확인: `atlas.mkhz.kr`을 브라우저에 입력해 정상 로드되는지, 자물쇠 아이콘(HTTPS)이 뜨는지 확인

## 4. 공개 전 최종 체크 (지시서 그대로)

실제 브라우저에서 **두목님이 직접** 확인해야 한다 (이 작업 환경은 cartocdn.com이 네트워크 차단돼 있어 타일이 안 뜨고, 실제 렌더링을 볼 수 없다):

- [ ] 지도 정상 로드 (타일이 뜨는지, 마커가 찍히는지)
- [ ] 검색 정상 동작 (예: "김구" 검색 → 결과 표시 → fitBounds)
- [ ] 검색 해제 → 초기 화면 정상 복귀
- [ ] 연도 슬라이더 이동 정상
- [ ] 점 클릭 시 팝업 카드 정상 표시
- [ ] 모바일 화면에서 검색바·범례·레이어 안 겹치는지
- [ ] 빈 화면(흰 화면/완전 깨짐) 없음
- [ ] 임의 경로 접속 시 404.html이 제대로 뜨는지 (예: `atlas.mkhz.kr/asdf`)
- [ ] 새로고침 시 정상 (현재 슬라이더 위치는 초기화되는 게 정상 동작 — SPA 라우팅이 없으므로 연도/검색 상태는 URL에 남지 않는다. 이는 의도된 동작이다, 6장 참고)
- [ ] 브라우저 개발자도구 콘솔에 에러 없는지
- [ ] HTTPS 자물쇠 정상

5. **공식 공개** — SNS 공유, 검색엔진 노출 시작

## 6. 향후 URL 전략 (결정만, 구현 없음 — 이번 작업 범위 제외)

이번 작업에서는 라우팅 구조(`/era`, `/person`, `/event` 등)나 SPA 전환을 만들지 않는다. 대신 **현재의 단일 페이지 구조를 유지하면서도 나중에 자연스럽게 확장할 수 있는 URL 형태**만 미리 정해둔다.

| 시점 | URL 형태 | 의미 |
|---|---|---|
| 지금 | `atlas.mkhz.kr` | 기본 진입, 초기 시야 |
| 향후 | `atlas.mkhz.kr/?year=1919` | 특정 연도로 진입 (슬라이더를 그 값으로 맞춤) |
| 향후 | `atlas.mkhz.kr/?search=김구` | 특정 검색어로 진입 (검색을 자동 실행) |

**왜 쿼리 파라미터(`?year=`, `?search=`)인가**: 현재 이 사이트는 페이지 이동이 없는 단일 HTML(정적 파일 하나)이다. `/era`, `/person` 같은 경로 기반 라우팅을 도입하려면 SPA 라우터(History API)나 서버 사이드 라우팅이 필요한데, 이건 이번 작업 범위에서 명시적으로 제외됐다. 반면 쿼리 파라미터는 **같은 `index.html` 한 장 안에서, `window.location.search`를 읽어 초기 상태(연도·검색어)만 다르게 시작**하는 방식이라, 지금 구조를 전혀 바꾸지 않고도 나중에 한 번에 구현할 수 있다.

**구현 시점에 할 일** (지금은 만들지 않음): `app.js`의 `window.onload` 안에서 `new URLSearchParams(location.search)`로 `year`/`search` 값을 읽어, 있으면 슬라이더를 그 연도로 맞추거나 `runSearch()`를 자동 호출하면 된다. 마커 클릭 시 또는 검색 시 `history.replaceState`로 현재 상태를 URL에 반영하는 것도 같은 방식으로 나중에 붙일 수 있다 — 지금 코드 구조를 바꿀 필요가 없다는 점이 이 전략을 선택한 이유다.

## 7. 이번 작업에서 제외한 것 (다음 단계로 미룸)

- 라우팅 구조(`/era`, `/person`, `/event`, `/place`, `/explore`), SPA 전환
- 세계사로의 데이터 구조 확장
- 검색 인덱싱 고도화, DB 도입
- 검색어·연도·클릭 단위의 GA 커스텀 이벤트 트래킹
- OG 이미지(실제 지도 스크린샷)

이번 공개는 완성이 아니라, 역사가 계속 자라날 수 있는 첫 공개 버전이다.
