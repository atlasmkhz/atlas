## 1. 근본 원인 진단

전 단계에서 데이터(`content.hero`/`content.gallery`)는 정상적으로 채워졌지만,
**`js/renderer.js`의 `popupHtml()` 함수가 `e.image`(v1의 단일 필드, 사용 0건)만
읽고 `content` 필드 자체를 전혀 참조하지 않았던 것**이 화면에 이미지가 안 보이던
유일한 원인이었습니다.

1단계 ①②(데이터/파일 검사) 결과:
- `content.hero`/`content.gallery`가 참조하는 이미지 경로: **166건 전부 실제
  파일 존재 확인** (누락 0건)
- 확장자: 166건 전부 `.webp`로 통일, 대소문자 불일치 0건
- 즉 데이터와 자산 자체에는 문제가 없었고, ③ 렌더러 쪽 문제로 확정.

## 2. 1단계 — 렌더러 수정 결과

`js/renderer.js`의 `popupHtml()`을 수정했습니다:
- `content.hero`를 우선 사용하고, 없을 때만 `e.image`(구버전)로 폴백 — 기존
  동작과 100% 하위호환.
- `content.gallery`를 새 영역(`.pop-gallery`)으로 추가, 관련 사건과 관련 영상
  사이에 배치(media-policy.md 카드 순서 규칙 준수).
- 이미지 로드 실패 시 `onerror`로 해당 영역을 `display:none` 처리 — **콘솔에
  JS 에러를 추가로 발생시키지 않고, 빈 칸이나 깨진 아이콘도 남기지 않습니다**
  (4번 검증 결과 참고).
- `css/style.css`에 `.pop-ai-note`(AI 표기), `.pop-gallery`/`.pop-gallery-item`
  (갤러리 그리드) 스타일을 새로 추가. 기존 클래스·디자인 톤은 변경하지 않았습니다.

## 3. 2단계 — 25명 인물 연결 결과

전용 person 카드도 없고 동명 사건 카드도 없던 25명을 새 규칙(①인물명 포함 사건
→ ②관련도 최고 사건 → ③여러 명이면 hero 유지+gallery 추가)에 따라 19개 기존
사건에 연결했습니다.

| 처리 방식 | 인물 수 |
|---|---|
| 제목에 이름이 직접 포함된 사건에 연결 | 5명 (고종, 이토 히로부미, 전명운, 장인환, 조명하) |
| 관련도가 가장 높은 사건에 연결(단일 등장) | 12명 |
| 관련도가 가장 높은 사건에 연결(여러 사건 중 선택) | 8명 |
| **합계** | **25명 (100% 연결 완료)** |

기존에 hero가 있던 13개 사건(을사늑약, 갑신정변, 청산리 등)은 **사건 대표
이미지를 그대로 유지**하고 인물 사진은 전부 `gallery`로 추가했습니다(교체 없음).
hero가 없던 6개 사건(통리기무아문 설치, 무오독립선언 등)에는 인물 사진을 hero로
사용했습니다.

## 5. 4단계 — 데이터 구조 정리 결과

일괄 보완했습니다(`is_ai` 값을 근거로 `archive`/`generated`를 채움). 최종적으로

```js
  gallery:[...]

```

확인됐습니다.

## 9. 실패 목록 / 알아둘 점

1. **이 작업 환경(컨테이너)에서는 Leaflet 지도 자체를 실제로 띄울 수 없습니다**
   — CDN(`cdnjs.cloudflare.com` 등)이 네트워크 화이트리스트에 없어 차단됩니다.
   이건 이번 작업으로 생긴 문제가 아니라 원래부터 있던 환경 제약이며,
   `docs/README.md`에도 이미 기록되어 있습니다. 그래서 검증은 지도 없이
   `popupHtml()`이 만드는 카드 HTML만 독립적으로 렌더링해 확인했습니다 —
   카드 자체의 마크업과 이미지 표시 로직은 지도와 무관하게 동일하게 동작하므로
   검증 결과는 유효합니다. 실제 지도가 있는 환경(박효순님의 배포 사이트)에서
   한 번 최종 확인을 권장합니다.
2. **영상 썸네일(유튜브) 로드도 같은 이유로 이 환경에서는 깨집니다** — 이는
   이번 작업 범위(이미지 hero/gallery) 밖의 기존 기능이라 손대지 않았습니다.
3. **나머지 198건의 사건 + 다수 인물 카드는 여전히 이미지가 없는 빈 카드**입니다
   — 이는 실패가 아니라 아직 사진을 못 받은 정상 상태입니다(이번 작업은 표시
   안정화이지 신규 수집이 아니므로).
