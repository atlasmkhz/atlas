# 타일 비교 평가 — legacy(가리개) vs world(no-label + overlay)

## 검증 방법과 한계 (먼저 밝힘)

이 평가는 다음 두 가지를 결합한 결과다.

1. **출처 신뢰도 확인 (완료)**: `dark_nolabels`가 실제로 존재하는 공식 엔드포인트인지 웹 검색으로 확인했다. Carto 공식 GitHub 저장소(`CartoDB/basemap-styles`)의 `docs/basemap_styles.json`과 README가 `dark_nolabels`를 `light_all`, `dark_all` 등과 동일한 위치에 정식 스타일로 나열하고 있다. URL 패턴(`https://{s}.basemaps.cartocdn.com/{style}/{z}/{x}/{y}{scale}.png`)도 공식 문서와 일치한다. 즉 **"이런 타일이 존재한다"는 사실 자체는 검증됨.**

2. **실제 타일 응답 확인 (미완료, 환경 한계)**: 이 작업 환경의 `bash_tool` 네트워크가 `cartocdn.com`을 차단하고 있고, `web_fetch` 도구도 검색 결과에 정확히 등장하지 않은 임의 URL(타일 PNG)은 가져올 수 없는 정책이라, **dark_nolabels 타일을 실제로 받아 "정말 라벨이 없는지" 픽셀 단위로 확인하지 못했다.**

따라서 이 문서는 "확정된 결론"이 아니라 **"코드는 준비되어 있고, 사용자가 실제 브라우저에서 `compare.html`을 열면 양쪽을 나란히 비교할 수 있다"**는 상태를 보고하는 것이다. 아래 평가 항목 중 ⚠ 표시는 실제 화면 확인이 필요한 부분이다.

---

## 비교 대상

| | A. legacy (현행) | B. world (제안) |
|---|---|---|
| 타일 | `dark_all` (라벨 포함) | `dark_nolabels` (라벨 없음) |
| 동해 표기 처리 | 위경도 사각형으로 일본해 라벨 위치를 추정해 가림 | 해당 위치에 라벨 자체가 타일에 없으므로 가릴 필요 없음 |
| 추가 라벨 | 동해, 독도 2개만 직접 렌더링 (도시명은 타일에 이미 있음) | 동해, 독도, 서울, 도섴 4개 — 도시명도 타일에 없으므로 직접 그려야 함 |

프로토타입(`chronicle_prototype/compare.html`)은 두 지도를 화면 좌우에 나란히 두고 줌/이동을 동기화하며, `dark_nolabels` 타일의 `load`/`tileerror` 이벤트를 콘솔에 자동 기록해 접근 가능 여부를 진단한다.

## 평가

### 성능
- **이론적 예상**: 동일한 Carto 인프라이므로 타일 1장당 응답 속도는 `dark_all`과 `dark_nolabels` 사이에 의미 있는 차이가 없을 것으로 예상된다 (라벨 렌더링은 타일 생성 시점에 이미 끝나 있고, 우리가 받는 건 완성된 래스터 이미지이기 때문).
- **legacy의 추가 비용**: `seaLabelMask`(L.rectangle 1개)는 거의 무비용.
- **world의 추가 비용**: 도시 라벨이 늘어날수록 `L.marker`(divIcon) 개수가 늘어난다. 동해/독도/서울/도쿄 4개 수준에서는 무시할 수 있지만, "모든 지명"으로 확장하면(roadmap.md가 말하는 최종 형태) 마커 수가 수백~수천 개가 될 수 있어 이때는 `renderer.js`의 declutter 로직과 함께 성능을 재검토해야 한다.
- ⚠ **확인 필요**: `compare.html`의 `controls` 영역에 타일셋 로드 시간이 표시된다. 두 방식의 실측 차이를 직접 비교할 것.

### 가독성
- **legacy**: 가리개가 배경색과 정확히 일치하지 않으면 사각형 경계가 도드라져 보인다 (이전에 사용자가 스크린샷으로 지적했던 문제 — 좌표는 맞았으나 색상이 살짝 어긋났던 사례 있음).
- **world**: 라벨이 원천적으로 없으므로 경계 부자연스러움 자체가 발생하지 않는다. 다만 지금처럼 4개 라벨만 그리면, 그 외 지명(예: 평양, 부산, 만주 도시들)이 화면에서 완전히 사라진다 — 이건 가독성 **저하** 요인이다. 우리 데이터의 사건 위치(평양, 만주, 연해주 등)를 이해하려면 지명이 필요한데, 지금 프로토타입은 "동해/독도/서울/도쿄" 4개로 한정했기 때문에 실제 적용 시에는 사건이 몰린 주요 도시 10~20개 정도를 추가로 그려야 실사용 가능한 수준이 될 것이다.

### 유지보수
- **legacy**: 가리개 좌표가 추정값이라 Carto가 타일 스타일을 업데이트하면(폰트 크기, 라벨 위치 변경 등) 다시 추정하고 조정해야 한다. 검증 수단이 마땅치 않다는 게 가장 큰 약점.
- **world**: 라벨이 타일에 없으므로 Carto 쪽 변경에 영향받지 않는다. 다만 "어떤 지명을 어디에 그릴지"를 우리가 전부 관리해야 하므로, 지명 데이터셋(이름, 좌표, 표시 우선순위)을 따로 만들어 유지보수해야 한다. 단기적으로는 legacy가 손이 덜 가지만(이미 만들어져 있으므로), 장기적으로는 world가 더 안정적이다.

### 확대 품질
- 동일한 Carto 인프라의 같은 줌 레벨 체계(`maxZoom:11`)를 쓰므로, 타일 자체의 해상도·선명도는 두 방식이 같을 것으로 예상된다.
- ⚠ **확인 필요**: `dark_nolabels`가 정말로 `dark_all`과 동일한 지리 데이터(도로, 국경선, 음영 등)를 쓰는지, 혹은 라벨을 빼면서 다른 디테일도 함께 빠졌는지는 실제 화면에서 비교해야 안다.

## 종합 판단

| 기준 | 우위 |
|---|---|
| 성능 | 비슷할 것으로 예상 (미확정) |
| 가독성 (동해 표기만) | **world 우위** — 가린 게 아니라 원천 부재 |
| 가독성 (전체 지명) | **legacy 우위** (현재 프로토타입 기준) — world는 지명 데이터셋 구축 전까지 정보 손실 |
| 유지보수 | **world 우위** (장기) — Carto 업데이트에 영향받지 않음 |
| 구현 비용 | **legacy 우위** (이미 존재함) |

**결론**: world 방식이 동해 문제에 대한 구조적 해결이라는 점은 명확하다. 그러나 "동해/독도/서울/도쿄 4개만 그린 프로토타입"을 그대로 실서비스에 적용하면 다른 지명이 전부 사라지는 가독성 후퇴가 생긴다. 따라서 전환은 다음 조건이 갖춰진 뒤 진행해야 한다:

1. 현재 데이터(197개 사건)에 등장하는 주요 지명을 추출해 라벨 데이터셋 작성 (최소 20~30개 도시/지역명)
2. 실제 브라우저에서 `dark_nolabels` 타일 접근이 정상적으로 되는지 확인 (이 환경에서는 못함 — 사용자 또는 다른 환경에서 확인 필요)
3. 위 1, 2가 확인되면 `seaLabelMask` 제거하고 `MAP_MODE.current`를 `'world'`로 전환

지금 단계에서는 **`seaLabelMask`를 제거하지 않는다.** 성공 조건(완료 조건: "표시 자체가 발생하지 않는 구조")이 충족됐다고 확정할 근거가 아직 부족하기 때문이다.
