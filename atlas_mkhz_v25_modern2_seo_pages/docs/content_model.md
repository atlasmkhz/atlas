# content_model.md — 카드 UI 구조 재설계 (설계 문서, 미적용)

`schema_v2.md`가 데이터 쪽 설계라면, 이 문서는 그 데이터를 화면에 어떻게 배치할지의
설계다. 이번 단계에서는 와이어프레임 수준의 구조만 정의하고, 실제 CSS/JS 구현은
하지 않는다(7단계 "이번 작업 금지" 참고).

---

## 1. 현재 구조 (v1) 진단

`js/renderer.js`의 `popupHtml(e)` 함수가 만드는 현재 카드:

```
[대표 이미지]          ← e.image, 현재 0건 사용 (있으면 표시, 없으면 숨김)
[유형 배지]
제목
날짜 · 장소
[지속기간 배지]
[관련 인물]
개요 (summary_ko 전체, 길이 제한 없음)
[관련 사건 — 아코디언]
[관련 영상 — 1개, 썸네일+링크]
```

문제점:
- 이미지가 1개뿐이라 갤러리 개념이 없다. `e.image`가 있으면 칸 하나, 없으면
  그 자리가 통째로 사라진다 — "사진 3장이 들어오면 어떻게 되는가"라는 질문에
  현재 구조는 답할 수 없다.
- 영상도 1개뿐(`e.video`)이고, `related_content`로 들어온 영상(7건)은 또 다른
  자리(아래쪽)에 별도로 렌더링된다 — 같은 "영상"인데 두 군데에 다른 모양으로
  나타날 수 있는 구조다.
- 문서/지도/음성은 자리 자체가 없다.
- "개요"와 "상세 설명"이 분리되어 있지 않다 — `summary_ko` 하나가 길든 짧든
  전부 한 블록으로 나온다.

## 2. 신규 카드 구조 (v2) — 7개 영역

지시서의 ①~⑦ 영역을 그대로 채택하되, 각 영역이 데이터가 없을 때 어떻게
동작해야 하는지(=6단계 검증)를 영역별로 함께 정의한다.

```
┌─────────────────────────────┐
│ ① 대표 영역                  │
│   [대표 사진 hero]            │
│   유형 배지                  │
│   제목 (title_ko)             │
│   부제 (subtitle)             │
│   시기 · 장소 · 유형           │
├─────────────────────────────┤
│ ② 개요                      │
│   summary_short (3~5줄)        │
├─────────────────────────────┤
│ ③ 상세 설명                  │
│   배경 / 전개 / 결과           │
│   (= 기존 summary_ko 전체)      │
├─────────────────────────────┤
│ ④ 인물                      │
│   관련 인물 (people)           │
├─────────────────────────────┤
│ ⑤ 사건                      │
│   관련 사건 (connections)       │
├─────────────────────────────┤
│ ⑥ 콘텐츠                    │
│   사진 그리드 (gallery)         │
│   문서 (documents)             │
│   영상 (videos)               │
│   지도 (maps)                 │
│   [음성 audio는 ⑥ 하단 또는    │
│    ③ 안에 인라인 플레이어]       │
├─────────────────────────────┤
│ ⑦ 참고                      │
│   출처 (sources)              │
└─────────────────────────────┘
```

### 영역별 데이터 소스와 "없을 때" 동작

| 영역 | 데이터 소스 | 데이터 없을 때 |
|---|---|---|
| ① 대표 영역 | `content.hero`, `title_ko`, `subtitle`, `year`/`month`/`day`, `place_ko`, `type` | hero 없으면 그 자리에 유형색 배경의 플레이스홀더(또는 완전히 생략 — 6단계에서 결정 필요). 제목·시기·장소는 항상 존재(필수 필드)하므로 빈 카드는 없음 |
| ② 개요 | `summary_short` | 없으면 `summary_ko`에서 첫 1~2문장을 잘라 자동 생성(fallback). 완전 공백 카드 방지 |
| ③ 상세 설명 | `summary_ko` (현재 그대로) | 항상 존재(v1 필수 필드라 빈 경우 없음) |
| ④ 인물 | `people` | 빈 배열이면 영역 자체를 숨김(현재도 동일 동작) |
| ⑤ 사건 | `connections` | 빈 배열이면 영역 자체를 숨김(현재도 동일 동작) |
| ⑥ 콘텐츠 | `content.gallery/documents/videos/maps/audio` | **모두 비어 있으면 ⑥ 영역 전체를 숨긴다.** 일부만 있으면(예: gallery만) 그 하위 탭/섹션만 표시 |
| ⑦ 참고 | `sources` | 거의 항상 1개 이상 존재(1차 작업에서 100% 보장됨) |

## 3. 영역 ⑥ "콘텐츠"의 하위 레이아웃 (사진 1000장 대응)

지시서의 완료 기준 — "사진 1000장, 영상 500개가 들어와도 구조 변경 없이
유지 가능"을 만족시키기 위한 설계.

### 3-1. 사진(gallery)
- 그리드 레이아웃, 한 사건당 갤러리는 **초기 노출 N장 + "더 보기"** 방식을
  권장(N은 구현 시 결정, 예: 6장). 1000장이 들어와도 DOM에 한 번에 1000개의
  `<img>`를 만들지 않고, 가시 영역에 들어올 때 추가 로드(지연 로딩/페이지네이션)하는
  방식이어야 한다 — "구조"가 감당해야 할 부분은 데이터 모델(배열, 개수 무제한)이고,
  "구현"이 감당해야 할 부분은 렌더링 성능(가상 스크롤/지연 로딩)이다. 이 문서는
  전자만 확정하고, 후자는 사진 작업 단계에서 실제 자산 개수를 보며 구현한다.
- 이미지 1개당 모델은 동일(`{id, url, alt, caption, credit, image_type}`) —
  사건당 1장이든 1000장이든 배열 안의 항목 구조가 달라지지 않는다.

### 3-2. 영상(videos)
- 영상은 사진보다 무거우므로 기본은 썸네일 그리드 + 클릭 시 재생(현재 `pop-video`
  방식과 동일한 인터랙션을 N개로 확장). 500개가 들어와도 마찬가지로 가시 영역
  지연 로딩 대상.

### 3-3. 문서(documents) / 지도(maps)
- 문서는 신문 지면처럼 가로가 긴 이미지가 많을 수 있어, gallery와 다른 썸네일
  비율(예: 정사각형이 아니라 세로로 긴 비율)을 권장. 구조적으로는 gallery와 동일한
  배열 모델이라 별도 스키마 변경 없이 CSS 클래스만 다르게 줄 수 있다.
- 지도는 클릭 시 별도 라이트박스 또는 기존 Leaflet 지도와 겹쳐 보여주는 방식
  검토 가능(예: "이 사건의 당대 지도를 현재 지도에 오버레이"). 이 상호작용은
  사진 작업 단계에서 실제 자산을 보고 결정 — 이번 단계에서는 "자리"만 확보.

### 3-4. 음성(audio)
- 인라인 오디오 플레이어 1~N개. ③ 상세 설명 본문 중간에 삽입하는 방식(예: 증언
  음성을 본문 인용 옆에 배치)과 ⑥ 콘텐츠 하단에 모아 배치하는 방식 두 가지를
  검토. 이번 단계에서는 결정하지 않고 두 옵션을 문서로만 남긴다.

## 4. 6단계 — UI 구조 검증 (질문별 답)

### Q1. 사진 3장을 추가해도 깨지지 않는가?
**Yes (설계 의도상).** `content.gallery`는 배열이라 1장이든 3장이든 1000장이든
같은 구조다. 다만 **현재 v1 렌더링 코드(`popupHtml`)는 `content` 필드를 전혀
읽지 않으므로, 지금 당장 3장을 넣어도 화면에는 아무것도 나타나지 않는다** — 이건
구조의 결함이 아니라 "아직 렌더링 코드를 마이그레이션하지 않았다"는 사실의
정확한 반영이다. 렌더링 코드 작업은 이번 단계의 범위 밖(7단계 금지 항목인
"콘텐츠 수집"은 아니지만, 지시서가 "구조 설계만 수행"이라 명시했으므로 실제
`popupHtml` 리팩토링도 다음 세션으로 분리하는 것이 안전하다고 판단했다 — 5번
"남은 이슈" 참고).

### Q2. 영상을 추가해도 깨지지 않는가?
**구조적으로는 Yes.** `content.videos`도 배열이라 개수 제한이 코드에 없다. 단,
Q1과 동일한 이유로 현재는 렌더링 연결이 안 되어 있다.

### Q3. 모바일에서 접히는가?
**설계 의도는 Yes, 검증은 다음 단계.** 기존 `infoPanel.js`가 이미 모바일에서
중앙 모달, 데스크탑에서 우측 패널로 분기하는 구조를 갖고 있고(파일 상단 주석
참고), 새 카드 구조(①~⑦)도 그 컨테이너 안의 세로 스크롤 콘텐츠일 뿐이라 컨테이너
분기 자체에는 영향이 없다. 다만 ⑥ 콘텐츠 영역의 그리드(사진 여러 장)가 좁은
모바일 폭(infoPanel의 모바일 모달 너비)에서 몇 열로 보일지는 실제 CSS 작성 시
결정해야 하는 문제라, 이번 설계 문서에서는 "컨테이너 레벨 접힘은 기존 구조를
그대로 쓸 수 있다"까지만 확인하고 그리드 열 수 같은 세부는 다음 단계로 넘긴다.

### Q4. 텍스트만 있어도 읽히는가?
**Yes.** 표(2번)에서 보듯 ④⑤⑥은 데이터가 없으면 영역 자체가 숨겨지고, ①②③⑦은
v1에서부터 항상 채워지는 필수 필드 기반이라 콘텐츠 0개 상태(현재 343건 전체)에서도
카드가 빈 칸 없이 완전하게 보인다. 이는 현재 `popupHtml`이 빈 `image`/`video`를
처리하는 방식(필드가 없으면 그 블록 자체를 비움, 깨진 이미지 아이콘을 보여주지
않음)과 같은 원칙을 ⑥ 전체로 확장한 것이다.

## 5. 남은 이슈 (다음 세션 후보)

1. **`popupHtml()` 자체의 리팩토링은 이번 세션에서 수행하지 않았다.** 지시서가
   "구조 설계만 수행"을 요구했고, 실제 함수를 ①~⑦ 구조로 재작성하는 것은 코드
   변경 범위가 커서(또한 사진/영상이 실제로 0개인 상태에서 빈 갤러리 UI를 미리
   구현하는 것은 "보여줄 것 없는 UI를 먼저 만드는" 본말전도가 될 수 있다고
   판단했다) 다음 세션으로 분리하는 것을 권장한다. 단, `schema_v2.md`와 이 문서가
   합의되면 그 다음 세션에서는 코드 변경(렌더링 함수 분리, CSS 그리드 추가)을
   바로 시작할 수 있도록 설계를 구체적으로 작성해 두었다.
2. **갤러리 초기 노출 수(N), 지연 로딩 구현 방식**은 실제 평균 자산 개수를 보고
   정하는 것이 합리적이라 미정으로 남김.
3. **지도(maps) 콘텐츠의 인터랙션 방식**(라이트박스 vs 지도 오버레이)도 미정.
4. **음성(audio)의 배치 위치**(본문 인라인 vs 콘텐츠 영역 하단)도 미정.
