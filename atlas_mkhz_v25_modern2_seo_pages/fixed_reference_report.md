# 이미지 파일명 ASCII 전환 작업 리포트

## 배경 — 문제 원인 확정

Cloudflare Pages 배포 환경에서 1920년대 인물 카드(박헌영, 안창호, 이봉창 등) 이미지가
표시되지 않는 문제가 보고되었다. 로컬 검증 환경(Linux, 직접 압축 해제)에서는 동일한
데이터·이미지 경로가 항상 정상 작동했기 때문에, 데이터-파일 1:1 전수 대조로는 깨진
참조가 발견되지 않았다(0건).

남는 유력한 원인은 **한글 파일명의 유니코드 정규화 차이**다:
- macOS/Cloudflare 업로드 경로에서 한글이 NFD(자모 분리형)로 재인코딩되는 경우,
  데이터에 박혀 있는 NFC 문자열과 실제 서버상의 파일명이 바이트 단위로 달라져
  브라우저가 404를 받는다.
- `onerror` 핸들러가 이미지 영역 자체를 숨기도록 설계되어 있어(media-policy.md
  "플레이스홀더 금지" 원칙), 사용자 화면에는 에러 메시지 없이 "이미지가 원래
  없는 것처럼" 보인다 — 정확히 보고된 증상과 일치한다.

근본 해결을 위해 인코딩 문제 자체가 발생할 수 없도록, `assets/images/entity/` 내
한글 파일명 전체를 ASCII(영문 소문자 + 언더스코어)로 전환했다.

## 작업 범위

- 대상 디렉토리: `assets/images/entity/person/` (event/place 폴더는 이미 영문이라 대상 없음)
- 대상 파일 수: 75개 (person 74개 + gallery 1개)
- 변경 규칙: `person_<한글이름>_NN.webp` → `person_<로마자표기>_NN.webp`
  - 소문자만 사용, 공백 없음, 언더스코어(`_`)만 구분자로 사용
  - 로마자 표기는 가능한 경우 데이터의 `title_en` 필드에서 기존 표기를 추출해 일관성 유지
    (예: 손기정 → `sohn_kee_chung`, 이승만 → `syngman_rhee` — 이미 굳어진 표기 우선)
  - 나머지는 표준 로마자 표기 규칙 적용

## 수행 내역

1. `assets/images/entity/` 전체 스캔 → 한글 포함 파일 75건 식별 (event/place 0건)
2. 한글 인명 → ASCII 매핑 테이블 작성 (`rename_map.csv`)
3. 실제 파일 75건 모두 rename 수행 — 성공 75건, 실패 0건 (이동/삭제/내용 변경 없음, 순수 rename)
4. `data/*.js` 전체(63개 파일)에서 구 경로 문자열을 신규 경로로 치환
   - 총 97건 치환, 40개 데이터 파일에 걸쳐 분포 (한 인물이 여러 연도 카드에 등장하기 때문)
5. 전수 재대조: 데이터가 참조하는 158개 url 전체가 디스크 파일과 1:1 일치 확인 (깨진 참조 0건)
6. 로컬 HTTP 서버로 158개 이미지 URL 전체를 실제로 fetch — 200 응답 + `image/*` Content-Type
   100% 확인 (`image_integrity_report.md`)
7. Playwright로 실제 카드 렌더링 검증 (박헌영·안창호·이봉창·삼둔자전투 갤러리 등) — 정상 표시 확인

## 절대 준수 사항 (금지 항목 확인)

- 이미지 재인코딩: 하지 않음 — `shutil.move`로 순수 rename만 수행, 픽셀 데이터 무변경
- 이미지 내용 변경: 없음
- 폴더 이동: 없음 — 모든 파일이 원래 폴더(`assets/images/entity/person/`) 내에서 이름만 변경
- hero/gallery 구조 변경: 없음 — `content.hero`, `content.gallery` 필드 구조 그대로 유지,
  `url` 문자열 값만 교체
- 렌더러(js/renderer.js) 수정: 없음

## 잔존 한글 파일명

전수 스캔 결과 `assets/images/entity/` 전체에 한글 포함 파일명 0건. 모두 ASCII로 전환 완료.

## 권장 후속 조치

- 향후 Numbers → 이미지 추출 자동화 스크립트(작업지시서 v2.0의 `npm run import-images`)에
  "파일명 ASCII 변환" 단계를 표준 절차로 포함시킬 것을 권장한다. 이번처럼 한글 인명을
  파일명에 그대로 쓰는 방식은 로컬 환경에서는 문제없이 보여도, 배포 파이프라인(특히
  맥OS 클라이언트 → Cloudflare 업로드 경로)에서 인코딩 사고가 재발할 수 있다.
